Round 2 · Deeper Redesign · 2026-07-21

不延伸 v1 的方向，
换两个根本不同的思路

v1 给 01 和 04 的方案是"补丁式"的——加横滑、加搜索、加徽章。 这一轮换两个范式：01 把评价和作品图绑定（化妆师独有的可信度杠杆）， 04 把"列表"升级成"需要关注的客户视图"（让系统主动浮起风险）。

横滑评价卡 + CTA 主次
2-3 条 latest review 横滑。问题：和其他小程序的评价区无差异，缺化妆师独有的证据。

搜索 + 卡片加徽章
加搜索 + 过敏/服务徽章。问题：仍是数据库式列表，化妆师要主动找问题。
v2 · Case 04 新方向
经营视图 · 风险优先
"需要关注"section 主动浮起：过敏 / 流失风险 / 新客欢迎。列表降级为索引。
01R2
客户侧 · 首页 · 客户评价模块

作品×评价联动 — 化妆师独有的可信度

其他行业（餐饮/美发/医美）的评价只能是"文字+星"。化妆师不同——每条评价背后都有一件具体作品。 把"作品缩略图 + 客户评价"同框展示，让口碑不再是空的形容词，而是有视觉证据的具象推荐。

Direction Pivot · v1 → v2
v1 做"横滑评价卡 + 标签云简化"——和其他小程序没区别。 v2 换思路：每条评价都带上对应作品的缩略图 → 客户看到的不只是"小红说好"，而是"小红做的这个新娘妆 + 小红说好"。 这是化妆师品类独有的 social proof 杠杆，餐厅/美发店都抄不走。

1
评分分布条 — 比"4.8 / 5.0"更有说服力
一个数字 4.8 看起来漂亮但不可信（5 分制容易虚高）。分布条直接告诉客户： "126 条里 102 条是 5★"（81%）——这是无法造假的证据。 权重 5★ 用金色渐变，4★ 中度，3★ 弱化。
数据来源：reviews 集合 rating 字段聚合，云函数已有
2
作品缩略图 + 评价 = 化妆师独有的可信度
每条 featured review 旁配 对应作品的缩略图（48×48 圆角方块）。 客户点缩略图可跳到作品详情——形成"看作品→看口碑→看作品→预约"的闭环。

餐厅评价只能配菜品图（菜品 ≠ 厨师手艺）；化妆师的"作品"就是手艺本身，这是品类独占的杠杆。
实现：review 文档已有 booking_id → booking.service_id → service.works 关联，1 跳查询
3
服务品类徽章 — 给评价加场景
"新娘妆 / 订婚妆 / 日常妆"小徽章紧贴星级。客户做的是订婚妆时，会优先看订婚妆的评价。 这是场景化匹配，比笼统的"4.8 ★"更精准地解决"这个化妆师适合我的场合吗"。
4
只显示 2 条 featured + 查看全部
v1 想做横滑 2-3 条但没说服力。v2 只显示 2 条 featured（带作品图的）， 剩余 124 条通过 查看全部 126 条 › 跳到独立列表页。 用稀缺性换冲击力——两条精选 > 五条平庸。
需要新增：reviews 列表页（独立于 admin 后台），简单列表+筛选
5
CTA 主次保留 v1 方案
预约按钮（主）38px 渐变 + 阴影；微信联系（次）26px outline + "（先问问）"暗示 fallback。 这部分 v1 已经做对了，v2 不动。
Trade-off · v1 vs v2
v1 横滑评价卡
+ 实现成本极低，纯前端
+ 可显示更多条评价（3-5）

− 和其他小程序评价区无差异
− 评价是孤立文字，缺视觉锚
− "4.8 分"仍然只是个数字
v2 作品×评价联动
+ 化妆师品类独占，竞品抄不走
+ 评分分布条提供不可造假证据
+ 形成"作品→口碑→作品"闭环

− 需要云函数 join reviews↔works（~30min）
− 只显示 2 条 featured，需独立"查看全部"页
04R2
化妆师后台 · 客户管理

经营视图 — 让"风险"主动浮起来

客户列表 v1 是数据库式：所有客户平铺，靠化妆师自己翻找问题。 v2 改成 CRM 视图：系统主动算出"需要关注"的三类客户（过敏/流失/新客）， 置顶成 section；普通列表降级为底层索引。

Direction Pivot · v1 → v2
v1 做"加搜索 + 卡片加徽章"——化妆师仍要主动浏览找问题。 v2 换思路：系统主动把风险浮起来 → 「需要关注」section 自动聚合：有过敏史的（红）/ 60+ 天未到店的（金）/ 14 天内新客（蓝）。 化妆师打开页面第一眼看到的不是"全部 55 个客户"，而是"今天该回访的 3 个"。

1
「需要关注」section — 把数据库变成助手
v1 的列表是被动数据库：化妆师要自己翻找"今天该联系谁"。 v2 顶部置顶 3 类需要主动行动的客户（过敏/流失/新客）， 每张卡是一句话+一个动作。化妆师打开页面第一眼就看到"今天该做什么"。
类比：邮箱的"重要"标签 vs 全部收件箱
2
过敏 alert（红）— 安全闸门前置
有过敏备注的客户 始终置顶，红色边框+图标。 化妆师即便不点开详情，也知道"接这个客户单前要再确认过敏"。 这是 v1 完全缺失的安全层（v1 把过敏埋在卡片第二行徽章里）。
实现：customer_notes.allergy 字段非空 → 触发置顶
3
流失风险（金）— 复购唤醒
"95 天未到店 + 曾完成 5 次"是高价值流失客户。系统自动按 days_since_last > 60 AND completed_count >= 2 聚合。 化妆师点 回访 › 可直接发订阅消息或微信联系。

这是 v2 的核心商业价值——客户档案的最终目的是复购，不是档案本身。
数据源：bookings 完成时间 + customers.completed_count
4
新客欢迎（蓝）— 转化为回头客
"首单 7 天内 + 仅 1 次完成"是新客。 化妆师点 欢迎 › 可主动发"第二次预约享 9 折"等激活消息。 把"新客 → 回头客"的转化做成显式动作。
注：PROJECT.md 写"客户管理/会员系统 out of scope"，但欢迎话术≠会员系统
5
普通列表 — 降级为底层索引
需要关注 section 下面才是「全部客户」普通列表。 v1 把所有客户平铺——化妆师要自己找。v2 让普通列表只承担"按图索骥"的功能： 已知客户名时通过搜索找；不知道时通过「需要关注」主动发现。
6
列表卡片 — 极简化、信息密度优先
单卡只显示 4 件事：头像 / 昵称+标签 / 最近服务+时间 / 完成次数。 过敏客户加红色 ⚠ 角标，流失客户加金色 ↻ 角标—— 列表里也能看到风险状态，不用切回 section。
Trade-off · v1 vs v2
v1 搜索 + 卡片加徽章
+ 实现成本极低（1.5h）
+ 不需要新计算逻辑

− 仍是被动数据库
− 化妆师要主动找问题
− 过敏只在卡片里，不抢眼
− 没用上客户档案的复购价值
v2 经营视图 · 风险优先
+ 主动浮风险，化妆师第一眼看到该做什么
+ 过敏安全前置
+ 把客户档案变成复购工具
+ 列表里也能看风险角标

− 需新增云函数聚合（~2h）
− 三类规则需手动调（60天/14天阈值）
Round 2 Verdict

这两个新方向
都用上了项目独有的资产

v1 的方向是"补丁"——任何小程序都能做。 v2 让首页用上"作品图"这个独占资产（餐厅抄不走）， 让后台用上"客户档案"的真正价值（复购唤醒而不是档案本身）。
